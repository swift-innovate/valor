# Codex Analysis

## Findings

1. Critical: director authorization is trivially bypassable via a spoofed request header. `getRequestRole()` falls back to `X-VALOR-Role` whenever there is no authenticated session, and `requireDirector()` accepts `"director"` or `"system"` from that source with no signature or shared secret check. Any unauthenticated client can therefore create users, dispatch missions, or delete divisions by sending the header. See [src/auth/index.ts:18](/mnt/g/projects/sit/valor-engine/src/auth/index.ts#L18), [src/auth/index.ts:28](/mnt/g/projects/sit/valor-engine/src/auth/index.ts#L28), [src/api/users.ts:15](/mnt/g/projects/sit/valor-engine/src/api/users.ts#L15), [src/api/missions.ts:162](/mnt/g/projects/sit/valor-engine/src/api/missions.ts#L162), [src/api/divisions.ts:70](/mnt/g/projects/sit/valor-engine/src/api/divisions.ts#L70), [src/api/agents.ts:44](/mnt/g/projects/sit/valor-engine/src/api/agents.ts#L44).

2. Critical: several state-changing endpoints have no authorization at all, even before considering the header bypass above. Examples include approving/rejecting/revoking agent cards, mutating agents, creating/updating divisions, queueing missions, processing AARs, and mutating personas/artifacts/comms. That means an anonymous caller can directly alter core state and onboarding flow. See [src/api/agent-cards.ts:80](/mnt/g/projects/sit/valor-engine/src/api/agent-cards.ts#L80), [src/api/agent-cards.ts:94](/mnt/g/projects/sit/valor-engine/src/api/agent-cards.ts#L94), [src/api/agent-cards.ts:110](/mnt/g/projects/sit/valor-engine/src/api/agent-cards.ts#L110), [src/api/agents.ts:82](/mnt/g/projects/sit/valor-engine/src/api/agents.ts#L82), [src/api/agents.ts:113](/mnt/g/projects/sit/valor-engine/src/api/agents.ts#L113), [src/api/divisions.ts:35](/mnt/g/projects/sit/valor-engine/src/api/divisions.ts#L35), [src/api/divisions.ts:62](/mnt/g/projects/sit/valor-engine/src/api/divisions.ts#L62), [src/api/missions.ts:149](/mnt/g/projects/sit/valor-engine/src/api/missions.ts#L149), [src/api/missions.ts:217](/mnt/g/projects/sit/valor-engine/src/api/missions.ts#L217), [src/api/personas.ts:43](/mnt/g/projects/sit/valor-engine/src/api/personas.ts#L43), [src/api/artifacts.ts:38](/mnt/g/projects/sit/valor-engine/src/api/artifacts.ts#L38), [src/api/comms.ts:16](/mnt/g/projects/sit/valor-engine/src/api/comms.ts#L16).

3. High: a predictable default director account is auto-created on first boot. On an empty database, startup seeds `director` / `valor` unconditionally. In a fresh deployment, that is a ready-made administrative backdoor until someone changes it. See [src/index.ts:120](/mnt/g/projects/sit/valor-engine/src/index.ts#L120) and [src/db/repositories/user-repo.ts:94](/mnt/g/projects/sit/valor-engine/src/db/repositories/user-repo.ts#L94).

4. High: the user API exposes password hashes in normal responses. `User` includes `password_hash`, and `listUsers()`, `createUser()`, and `updateUserRole()` are returned directly from the API without sanitizing that field. Even if access control were fixed, returning credential material is unnecessary and increases breach impact. See [src/db/repositories/user-repo.ts:6](/mnt/g/projects/sit/valor-engine/src/db/repositories/user-repo.ts#L6), [src/api/users.ts:18](/mnt/g/projects/sit/valor-engine/src/api/users.ts#L18), [src/api/users.ts:37](/mnt/g/projects/sit/valor-engine/src/api/users.ts#L37), [src/api/users.ts:60](/mnt/g/projects/sit/valor-engine/src/api/users.ts#L60).

5. Medium: the current working tree does not typecheck, so it is not in a releasable state. `npm run typecheck` fails on multiple files, including a missing required `attachments` field in chat creation and several repository calls that no longer satisfy the `DbRow` generic constraint. This is not cosmetic; it means the code and its data-access abstraction are out of sync. See [src/api/comms.ts:105](/mnt/g/projects/sit/valor-engine/src/api/comms.ts#L105), [src/db/adapter.ts:1](/mnt/g/projects/sit/valor-engine/src/db/adapter.ts#L1), [src/db/repositories/user-repo.ts:55](/mnt/g/projects/sit/valor-engine/src/db/repositories/user-repo.ts#L55), [src/db/repositories/session-repo.ts:29](/mnt/g/projects/sit/valor-engine/src/db/repositories/session-repo.ts#L29), [src/db/repositories/approval-repo.ts:76](/mnt/g/projects/sit/valor-engine/src/db/repositories/approval-repo.ts#L76), [src/callbacks/sigint-outcome.ts:19](/mnt/g/projects/sit/valor-engine/src/callbacks/sigint-outcome.ts#L19).

6. Medium: webhook dispatch failure handling is too slow and currently causes a confirmed test failure. `dispatchWebhook()` hardcodes a 10-second abort, while the failing test expects unreachable endpoints to fail within 5 seconds; the result is a timeout instead of graceful degradation. The timeout is also only cleared on the success path. See [src/dispatch/webhook.ts:95](/mnt/g/projects/sit/valor-engine/src/dispatch/webhook.ts#L95), [src/dispatch/webhook.ts:109](/mnt/g/projects/sit/valor-engine/src/dispatch/webhook.ts#L109), [tests/dispatch/dispatch.test.ts:262](/mnt/g/projects/sit/valor-engine/tests/dispatch/dispatch.test.ts#L262).
